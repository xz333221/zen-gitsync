## 1. CLI 入口层

### 1.1 `src/gitCommit.js` (456 行)

**职责**：`g` 命令的总入口，分发所有子命令到 utils。

**关键函数 / 模式**

- `createGitCommit({exit})`：核心提交流水线
  1. `git status` 拿原始输出
  2. `judgeUnmerged` 检测冲突直接 `process.exit(1)`
  3. 无变更 → 智能分发（要 pull / 要 push / 都不要）
  4. 有变更 → `execDiff` → `execAddAndCommit` → `judgeRemote` → `exec_push`
- `commitAndSchedule(interval)` + `setTimeout(..., interval+100)`：串行递归而非 setInterval，避免并发提交冲掉 index.lock
- `startCountdown`：用 `log-update` 做原地倒计时 + boxen 美化 + 1Hz 刷新
- `main()` 串行调用：`judgePlatform → 检 'ui' → handleConfigCommands → handleFileLockCommands → --cmd= 调度 → --interval 循环`
- `--cmd="…"` 同时支持 `--at=HH:MM`、`--at=ISO`、`--daily`、`--cmd-interval=N`，4 套调度策略（立即 / 一次性定点 / 每日循环 / 周期）
- 亮点：用 `cmdArg.match(/^--cmd=(['"]?)(.*)\1$/)` 正确处理 `=` 后含 `=` 的命令；用 `%%→%` 还原 Windows 路径占位符

**独特设计**
- `cmdArg.replace(/%%/g, '%')` 是给 Windows `%PATH%` 留的口子
- `--cmd-interval` 与 `--interval` 可以并行：定时 Git 提交 + 定时自定义命令
- 冲突文件状态直接拒，不做任何自动合并尝试（用户必须手动）

### 1.2 `src/config.js` (351 行)

**职责**：单一真相源 `~/.git-commit-tool.json` —— 维护全局 ui 状态 + 跨项目配置 + Lock 文件 + 最近目录。

**关键结构**

```
顶层结构 (loadConfig 时识别老格式自动迁移):
{
  theme, locale, models, ui: { layout, commandConsole, fileListViewMode, ... },
  projects: {
    [normalizedProjectPath]: {
      defaultCommitMessage, commandTemplates, customCommands, orchestrations,
      startupItems, isStandardCommit, skipHooks, autoQuickPushOnEnter, ...
    }
  },
  recentDirectories: string[]
}
```

**亮点**
- `getCurrentProjectKey()`：优先 `git rev-parse --show-toplevel`，否则降级到 CWD；Windows 下统一小写避免 `D:\` vs `d:\` 误判为不同项目
- `loadConfig` 同时维护 schema 兼容（缺 `projects` 当老扁平格式）+ 全局字段（models/ui 始终从顶层取，不让项目级覆盖）
- `saveConfig` 写入前自动 `backupConfigFileIfExists` → `.bak` 兜底
- `lockFile / unlockFile` 用 `path.normalize` 而非直接比较，Windows 分隔符不踩坑
- `recentDirectories` 自动去重 + 上限 12 + 最新在前

### 1.3 `src/utils/index.js` (1089 行)

**职责**：CLI 真正的"瑞士军刀" —— Shell 输出美化、Git 命令执行、命令历史、平台适配。

**核心 API**

| 函数 | 用途 |
|------|------|
| `execGitCommand(cmd, opts)` | 包装 `child_process.exec`，自动设置 `GIT_CONFIG_PARAMETERS="'core.quotepath=false'"`，捕获 stdout/stderr 后挂到 `error.stdout/stderr` |
| `execSyncGitCommand(cmd, opts)` | 同步版本，spinner 协同 |
| `coloredLog / tableLog / printTableWithHeaderUnderline` | cli-table3 + chalk 表格化输出；超 10 行 / 行长 200 字符自动截断 |
| `errorLog(cmd, content)` | boxen 红框 + FAIL 标题 |
| `registerSocketIO / addCommandToHistory / getCommandHistory` | 命令历史跨进程广播（`ioInstance.emit('command_history_update')`） |
| `parseCwdArg(argv)` | 严格匹配 `--path`/`--cwd`/`--path=`/`--cwd=`/`--path <v>`，避免误匹配 `--pathTo=...` |
| `getCwd()` | 优先解析 `--path/--cwd`，否则 `process.cwd()` |
| `judgePlatform()` | Windows 跑 `chcp 65001` + `git config --global core.autocrlf true` + `core.quotepath false`；其它平台设 `input` |
| `judgeUnmerged(statusOutput)` | status 输出含 `unmerged paths` 直接 exit(1) |
| `judgeRemote()` | `git remote update` + `rev-list HEAD..@{u} --count`，behind>0 自动 ff-only pull（拉不动时附彩色消息） |
| `exec_push / execPull / execDiff / execAddAndCommit` | 完整提交原子操作 |
| `execGitAddWithLockFilter()` | `git status --porcelain` → 路径解析 → 跳过 Lock 匹配 → `git add -- <shQuote(file)>`（单文件失败不阻断整批） |
| `checkAndClearGitLock()` | 检测 `.git/index.lock` 存在则 `unlink` 兜底 |
| `addScriptToPackageJson / addResetScriptToPackageJson` | `g addScript` 自动注入 `g:y` / `g:reset` 脚本 |

**亮点（最值得借鉴）**
- `calcColor(commandLine, line)`：按命令名 + 行首内容自动着色（`new file:` 红 / `modified:` 绿 / `deleted:` 红）
- `MAX_HISTORY_SIZE=100 / MAX_OUTPUT_LENGTH=5000` + `isStdoutTruncated` 标记 → 前端能区分"完整"和"截断"
- `execGitAddWithLockFilter` 路径匹配逻辑：
  - 相对路径 vs 绝对路径两路处理
  - `path.relative(gitRoot, absoluteLocked)` 兜底跨平台
  - 双向 startsWith：`file` ⊆ `locked` 也要跳过
  - 目录特例：目录下全锁才整目录跳过
  - `shQuote`：手写单引号转义而非依赖 shelljs（`'\\''`）
- 错误透传：把 `error.stdout/stderr` 挂到 error 对象上，上层 `e.stdout` 可读完整上下文

## 2. 后端服务层

### 2.1 `src/ui/server/index.js` (421 行)

**职责**：组装 + 启动整个 UI 后端。

**关键流程**
1. `app = express()` + `io = new Server(httpServer)`
2. `currentProjectPath / projectRoomId` 用 let 而非 const，切换目录时可改
3. 启动即 `await configManager.saveRecentDirectory(gitRoot || cwd)` 自动入栈
4. 用闭包注册路由工厂（每个路由文件暴露 `registerXxxRoutes({app, ...})`）—— 状态通过 getter/setter 回调传入，避免模块全局污染
5. `registerCurrentInstance` 走 `httpServer.once('listening', ...)` 触发，避免端口重试时 `await` 时序与实际 listening 不一致
6. `instanceRegistry.watch(...)`：fs.watch 跨进程变更 → 广播 `instances_changed` 给前端
7. SIGINT/SIGTERM 优雅 unregister + 清 watcher + 清 heartbeat
8. `recentPushStatus` 5s 缓存 + 10s 推送标记，避免 push 完立刻被前端轮询 `branchStatus` 显示 "ahead 1" 的视觉抖动

**亮点**
- 端口策略：`resolveStartPort()` 先 env，否则 [4000, 6000) 随机起点
- `startServerOnAvailablePort` 拿到 port 立刻 `savePortToFile` + `open()` 浏览器
- 每个路由工厂都接收自己关心的依赖（`getCurrentProjectPath`、`runningProcesses` 等），没引入 DI 容器但达到了类似解耦效果
- `isGitRepo` 启动时一次性探测；后续路由用 `getIsGitRepo()` getter 拿

### 2.2 `src/ui/server/utils/instanceRegistry.js` (270 行)

**职责**：跨进程"哪些 GUI 正在跑"注册表，文件位于 `~/.zen-gitsync-instances.json`。

**关键设计**

| 维度 | 实现 |
|------|------|
| 并发写 | 进程内 `enqueueWrite` Promise 链串行化 |
| 原子写 | `tmp + rename` |
| stale 判定 | `process.kill(pid, 0)` + `lastHeartbeat > 30s` |
| 跨进程推送 | `fs.watch` 100ms debounce → list() → 推 callback |
| 自愈 | 每 `STALE_MS/3` 主动 `list({pruneStale:true})`，即僵尸条目被覆盖 |
| 心跳 | 5s setInterval；`process.on('exit')` 只能同步关 watcher（kill -9 走心跳超时） |
| 加载 | `pruneInPlace` 同时做"进程存活 + 心跳新鲜"双判 |

**亮点**
- `isProcessAlive` 只信 `ESRCH/ENOENT`，`EPERM` 视为"进程存在但权限不足"也算 alive
- `register` 把 `pid: process.pid` 写入后由前端 InstanceSwitcher 拿来去重（"我自己也算一个实例"）
- `watch` 返回 `closeWatcher` 函数，闭包持有 `closed` 标志，重复触发都被吸掉

### 2.3 `src/ui/server/utils/pathGuard.js` (155 行)

**职责**：路径越界校验中间件。

**两段 API**
- `pathGuard(cwd, fields?, opts)`：Express 中间件工厂，从 `req.query/body[field]` 取值 → `ensureWithinCwd` → 挂到 `res.locals.safePath`
- `ensureWithinCwd(input, cwd, opts)`：纯工具函数，返回 `{safePath, realPath}` 或 null

**校验链**
1. `path.resolve(cwd, input)` 解析 `..`、相对路径
2. `path.relative(cwd, resolved)` 以 `..` 开头 → 越界
3. Windows 大小写不敏感：两端 lowercase 再 startsWith
4. `opts.realpath` 时 `fs.realpath` 防符号链接，realpath 后**再**走一遍 (1)-(3)
5. 文件不存在保持 `realPath: null`，不报错，让 caller 决定

**亮点**
- 测试文件 `pathGuard.test.js` 覆盖：相对、绝对、含 `..`、跨平台、符号链接
- `ensureWithinCwdSync` 纯同步版本，路径解析不需要 IO 的场景用

### 2.4 `src/ui/server/utils/shellQuote.js` (67 行)

**职责**：跨平台 shell 转义三件套。

| 函数 | 平台 | 行为 |
|------|------|------|
| `shQuote(s)` | sh / bash | 单引号包裹 + `'` → `'\''` |
| `psEscape(s)` | PowerShell | 替换 `$\"\` 为转义形式 |
| `appleEscape(s)` | AppleScript | 替换 `\` 和 `"` |

**亮点**：所有用到 shell 注入风险的地方（`terminal.js` 的 Linux gnome-terminal / osascript / PowerShell）都引这些工具，绝不手拼字符串。

### 2.5 `src/ui/server/utils/randomStartPort.js` (51 行)

**职责**：端口策略。
- `process.env.PORT` 有值 → env 模式
- 否则 `{min: 4000, max: 6000}` 区间随机起点

### 2.6 `src/ui/server/utils/startServerOnAvailablePort.js` (101 行)

**职责**：端口冲突时顺序重试 + 自动开浏览器。
- `httpServer.listen(port, () => ...)` → 失败时递增到 `port+1`
- `maxTries: 100` 兜底
- 成功则 `open(noOpen ? null : url)` + `savePortToFile(port)`

## 3. 后端路由层

### 3.1 `routes/exec.js` (273 行) — 通用 + 流式命令执行

**API**
- `POST /api/exec`：非流式
- `POST /api/exec-stream`：SSE 流式（`spawn(cmd, [], {shell:true, env:{...FORCE_COLOR:3, TERM:xterm-256color, CLICOLOR_FORCE:1}})`）

**亮点**
- 智能 GBK 判定：仅 Windows **且** `firstToken` 命中 `cmdBuiltins = ['dir','type','set','path','cd','md','rd','del','copy','move','ren']` **且**（不是 echo 或 echo 不带变量替换）才转 GBK
  - `echo "{{xxx}}"` 时变量替换是 UTF-8，不转
- `stderr` 智能降级：先 UTF-8 → 含 `�` 或无中文 → 试 GBK
- `runningProcesses.set(processId, ...)` Map 维护，支持 `/process/kill/:id` 取消
- 不用 `req.on('close')` 主动 kill，让进程自然结束 → 避免 `process.exit` 早触发

### 3.2 `routes/terminal.js` (318 行) — 跨平台开新终端

**三平台分支**
- **Windows**：拼 PowerShell 脚本 `Start-Process -FilePath "cmd.exe" -ArgumentList "/K", "..." -WorkingDirectory "..." -PassThru`，`spawn('powershell.exe', ['-NoProfile', '-Command', psScript])` 拿 PID
  - `splitArgs` 手写带引号的 argv 解析（不引 shelljs）
  - `isStartCommand` 识别 `start` 前缀（额外支持 start URL → Start-Process URL）
  - 所有参数走 `psEscape` 转义
- **macOS**：`osascript -e "tell application \"Terminal\" to do script \"...\""`，引号走 `appleEscape`
- **Linux**：`gnome-terminal -- bash -c '...'` + `xterm` 兜底

**亮点**
- `isPidAlive`：Windows 走 PowerShell `Get-Process -Id`，其它走 `process.kill(pid, 0)`
- 终端会话 Map + `/restart` + `/status?cleanup=true` 10s 保护期（新启动的进程 10s 内即便探测不到也不清）
- `killTerminalPid`：Windows `taskkill /pid /T /F` 杀进程树

### 3.6 `routes/workbench.js` (3188 行) — 工作台（核心亮点）

**这是整个项目最大、最复杂的路由**，分多个子模块。

#### 3.6.1 持久化模型
```
~/.zen-gitsync/
├── prompts.json          # 预置提示词模板（生成项目架构说明用）
├── tasks.json            # 任务 + 子任务 + 附件
├── jobs.json             # 执行日志（1.5s debounce + 终态强制 flush）
├── jobs-config.json      # 保留策略（maxCount=500, maxSizeMB=256）
├── ai-instruction.json   # 用户可编辑的「项目架构说明」指令
├── ai-subtask-instruction.json  # AI 拆分子任务的指令（zh/en 双默认）
├── ai-split-sessions/    # 对话拆分多轮会话存档
└── workbench-images/     # 附件（_task-<id> / <subId> 分子目录）
```

**写入策略**
- 全部走 `tmp + rename` 原子替换
- `writeJson` 包了一层
- 启动时 `hydrateJobs()` 把磁盘上 `running/pending` 强降级为 `error`（原进程已死）

#### 3.6.2 核心调度函数

| 函数 | 职责 |
|------|------|
| `runTaskQueue(task, repoPath, branch, opts)` | 顺序跑所有 sub，`fromIndex` 支持"从中间开始" |
| `runSingleSubtask(task, sub, repoPath, branch, priorOutputs, options)` | 跑一个 sub：拼 prompt + launchClaude + 解析 NDJSON + 收尾 |
| `collectPriorOutputsUpTo(task, endIdx)` | 收集前序 done sub 的输出摘要（每条限 2000 字符） |

**Prompt 拼装（runSingleSubtask）**
1. `interpolate(template, ctx)` 替换 `{{task.title}} / {{task.desc}} / {{sub.title}} / {{sub.desc}} / {{repo.path}} / {{branch}}` 6 个变量
2. 前序 sub 输出拼到 prompt 头部（用 `priorOutputs` 数组）
3. 附件合并：sub 自己 + task 共享，按 `absolutePath` 去重（`virtualSub.attachments === task.attachments` 引用复用是常见坑 → 已专门处理）
4. 文件路径列表拼到 prompt 末尾（让 claude 自取）

**Claude 进程管理**
- `launchClaudeInNewWindow(cwd, prompt, resumeSessionId)`:
  - Windows 显式走 `claude.exe`（避开 Node 23 拒绝 spawn .cmd/.bat + shell: true 切 `\n` 的坑）
  - `where claude` → 读 `.cmd` 内容 → 推断 `claude.exe` 真实路径
  - macOS/Linux 直接 `spawn('claude', args, {detached: true, unref})`
  - 通用参数：`-p <prompt>` `--input-format text` `--output-format stream-json --verbose --permission-mode bypassPermissions --dangerously-skip-permissions`
  - `resumeSessionId` 时加 `--resume <id>`（**必须放在 -p 之前**）
- `waitProcessExit(pid)`：每 1.5s `process.kill(pid, 0)` 探活，30 分钟兜底超时
- `cancelledJobs: Set`：cancel 路由把 jobId 加进去，`runTaskQueue` 退出循环后根据集合标 `cancelled` 而非 `done`

**流式 NDJSON 解析**
- `parseLines(channel, buf)` 按行切
- 非 stream-json 行 → 原样进 `output` 兜底
- `system.init` 事件抓 `session_id`（首次出现覆盖），用于后续 `--resume` 续接
- `assistant.text` → `output`；`assistant.thinking` → `thinking`
- 100MB 兜底（`.slice(-MAX_OUTPUT)` 保留末尾）
- **关键**：流式 publish 推增量 delta（`output.slice(prevLen)`），终态或重连才推全量

**三类执行入口**
- `POST /api/workbench/tasks/:id/run`：整批按序
- `POST /api/workbench/tasks/:id/run-from {startSubIndex}`：从指定 sub 开始（前序 done sub 自动填上下文）
- `POST /api/workbench/tasks/:id/run-simple`：type=simple 的任务，合成 virtualSub 复用 `runSingleSubtask`
- `POST /api/workbench/subtasks/:id/run`：单 sub 执行
- `POST /api/workbench/jobs/:id/continue`：基于 `claudeSessionId` `--resume` 续接（仅 simple 任务）

**3.6.3 AI 拆分子任务（最复杂模块）**

`POST /api/workbench/tasks/ai-split-subtasks`（SSE）

1. 读 prompt 模板（按 `Accept-Language` 选 zh/en 默认）→ `readSubtaskInstruction(req)`
2. 加载 `promptId` 对应的执行模板，作为"子任务会被这套模板包裹后送进 claude"的提示
3. 加载 task 附件，把图片读成 `data:image/...;base64,...` 一起发给 LLM
4. 拼 user prompt（标题 / 描述 / 附件 / 模板 / 项目上下文）
5. **5 维度分析要求**：真实目标 / 关键技术栈与边界 / 自然执行顺序 / 风险点 / 是否需要前置调研
6. **JSON 输出严格要求**：title/desc 引用必须用「」或『』，禁 ASCII 双引号
7. 流式调 LLM → SSE push `meta / thinking / content / done / error`
8. 多级 JSON 降级解析（见下文）

`POST /api/workbench/tasks/ai-split-preview`：同样拼装但不调 LLM，前端编辑后把 `customUserBlock` 传回 `ai-split-subtasks` 即可

`POST /api/workbench/tasks/ai-chat-split`：多轮对话拆分，每轮 `sessionId` 持久化到 `ai-split-sessions/<id>.json`

`parseSubtaskJson(content)` 4 级降级（核心亮点）
1. 剥 `<think>...</think>` 思考块（deepseek-r1 / QwQ 风格）
2. ```json / ``` / 平衡花括号 / 整段
3. 去 `//...` `/* */` 注释 + 尾随逗号
4. 平衡花括号扫描（手写括号深度扫描，识别字符串字面量包括 `'` `"`）
5. `reescapeUnescapedQuotes`：启发式转义字符串内部未转义 ASCII `"`（**实战最常见错误**：模型用 `"保存"` 打断外层 JSON）

### 9.1 后端 Socket 实时层 — `socket/registerUiSocketHandlers.js` (227 行)

**职责**：交互式命令实时双向流（`exec_interactive` 事件）。是 `/api/exec-stream` 的姐妹，但走 Socket.IO 而非 SSE，方便用户在前端 CommandConsole 反复 stdin。

**关键设计点**

| 维度 | 实现 |
|------|------|
| 子进程 | `spawn(cmd, [], {shell: true, cwd, env: {...color force, GIT_CONFIG_PARAMETERS, PYTHONUNBUFFERED}})` |
| 进程记录 | `runningProcesses.set(processId, {childProcess, command, startTime, directory, sessionId})` —— 与 `/api/exec-stream` 共用一个 Map |
| GBK 智能转换 | 仅 stdout: Windows + `cmdBuiltins` 前缀命中才 iconv.decode；`echo` 特殊：含 `{{`/`${`/`%` 时不强转（变量替换语义优先） |
| stderr 更保守 | 先 UTF-8 试解：含 `一-龥` 或不含 `�` → 用 UTF-8；否则再 GBK；GBK 还失败 → 退回 UTF-8（**避免乱码叠加**） |
| stdin | 事件名按 session 隔离：`interactive_stdin_${sessionId}` —— 多个 session 并行互不干扰 |
| stop | Windows: `taskkill /pid ${pid} /T /F`（杀进程树）；POSIX: SIGTERM → 2s 后 SIGKILL 兜底 |
| 错误处理 | `error` 事件单独路径（不重复 `close`）—— close 删 Map + 写历史；error 写历史 + emit `interactive_error` |

**亮点**
- per-session 事件名隔离：`interactive_stdin_${sessionId}` 让 N 个 session 并行能各自响应
- 颜色环境变量 5 件套（`FORCE_COLOR=3` + `NPM_CONFIG_COLOR` + `TERM=xterm-256color` + `COLORTERM=truecolor` + `CLICOLOR_FORCE=1`）保证 npm/node/git 的 ANSI 输出不被打成纯文本
- Python 用 `PYTHONUNBUFFERED=1` 防 stdout 缓冲造成的"卡住"假象

### 9.2 后端 Routes 细节

#### 9.2.1 `routes/exec.js` (273 行) — 通用 + 流式命令执行

- 两个端点：
  - `POST /api/exec`：非流式一次性返回 `{stdout, stderr}`
  - `POST /api/exec-stream`：SSE (`text/event-stream` + `data: ${JSON.stringify(...)}\n\n`)，前端可用 EventSource 消费
- **streamId pre-increment**：`nextProcessId` 在 `index.js` 里用 `++processIdCounter`（不是 `processIdCounter++`），首进程从 1 起；`processId=0` 在 `if (processId)` 真值检查里会被吞，是真实修过的 bug
- 头部 `X-Accel-Buffering: no` 显式禁 nginx 缓冲（虽然本地 server 用不到，但保留兼容）
- **不在 `req.on('close')` 里杀进程** —— 注释明确说"参考 git push 实现"，避免 abort 误杀自然结束的进程
- 进程结束后统一通过 `addCommandToHistory` 写历史，与 socket 路径共用同一个 Map

#### 9.2.2 `routes/terminal.js` (318 行) — 在外部终端中执行命令

**职责**：把命令 `open` 到操作系统原生终端（Windows Terminal / macOS Terminal.app / Linux gnome-terminal），而不是跑在 zen-gitsync 自己 spawn 的子进程里。

**三平台实现**

| 平台 | 启动方式 | 关键 API |
|------|---------|----------|
| Windows | `powershell.exe -NoProfile -Command "Start-Process ... -WindowStyle Hidden -PassThru"` | `psEscape()` 转义所有 `$ \" \`` |
| macOS | `osascript -e 'tell application "Terminal" to do script "cd … && cmd"'` | `appleEscape()` + 外层 `shQuote()` |
| Linux | `gnome-terminal -- bash -c "…"` 失败 fallback `xterm -e` | `shQuote()` |

**进程状态管理**
- `terminalSessions` Map: `{id, command, workingDirectory, pid, createdAt, lastStartedAt}`
- 启动即调 `Start-Process` 拿 PID 写回 → 后续 `/api/terminal-sessions/status?cleanup=true` 用 PowerShell `Get-Process -Id` 探活
- **10s 保护期**：`PROTECT_DURATION_MS = 10_000` —— 刚启动的进程即使探活失败也不删，给 PowerShell `Start-Process` 异步启动留时间（防止冷启动被误清）

**亮点**
- `splitArgs` 手写 Windows 命令行 tokenizer（双引号转义 + 空白分隔），不依赖 winston
- 三个转义函数各司其职，绝不混用（PS 变量 `$` 用反引号取消、AS 双引号用反斜杠取消、sh 单引号是语言层不可变的最稳形态）

#### 9.2.3 `routes/status.js` (67 行) — Git 状态 + 命令历史

### 9.6 前端 Composables

#### 9.6.1 `composables/useNetworkStatus.ts` (160 行) — OPT-5 全局网络监听

**核心思路**：patch 一次 `window.fetch`，拦截所有 5xx / 网络失败 → 顶部横幅提示。

**实现亮点**：
```ts
originalFetch = window.fetch.bind(window)  // 保留原引用

window.fetch = async function patchedFetch(input, init) {
  const response = await originalFetch!(input, init)
  if (!response.ok && response.status >= 500) {
    isOnline.value = false
    lastError.value = `HTTP ${response.status}`
    lastFailedRequest.value = { url, init }
  } else if (response.ok) {
    // 注意: 不重置 isDismissed —— 用户主动关闭的语义是"这次别再烦我"
    if (!isOnline.value) isOnline.value = true
    lastError.value = null
  }
  return response
}
```

**设计哲学**
- `start()` / `stop()` 配对调用，HMR 时 stop 还原 `window.fetch = originalFetch`，不污染环境
- `lastFailedRequest` 暂存最近失败请求 → `retryLast()` 一键重试
- `isDismissed` 与 `lastError` 分离：用户关掉横幅 ≠ 后端恢复（避免后台成功请求把横幅重新拉出来）
- 同时绑 `online` / `offline` 事件作为 fetch 拦截兜底
- `describeError` 对 `TypeError: Failed to fetch` 走 i18n key `'@NET:网络连接失败'`，其它直接读 `error.message`

**为什么不放在 main.ts？** 注释解释：放在 App.vue onMounted 调，能保证**只 patch 一次**（HMR 时 App.vue 重 mount 触发 start，stop 在 onBeforeUnmount）。放在 main.ts 没法 stop，会越 patch 越深。
